# 멀티모달 입력 계약 — 제품의 뇌를 모달리티와 분리하기

## 한 줄 요약
`rank.py`(제품의 뇌)는 이미 **모달리티 무관**이다. 글의 내용도, 그것이 텍스트인지
이미지인지 비디오인지도 보지 않는다. 그래서 미래에 이미지·비디오를 추가해도 *뇌는
손대지 않는다* — 각 모달리티 파이프라인이 동일한 후보 계약만 내놓으면 된다.

## 계약 — `sies/candidate.py`의 `Candidate`
랭커가 후보에서 읽는 것은 단 셋뿐:

| 필드 | 용도 | 비고 |
|---|---|---|
| `distance` | 관련성(코사인) | **정규화 임베딩**(코사인 공간)에서 와야 함. `cosine_from_l2`가 d²=2(1−cos)를 가정 |
| `timestamp` | 시간 활성도 | ISO 날짜. 없으면 중립(0.5) |
| `doc_path`/`title` | 볼륨 활성도(`newer_counts`) | 문서 정체성 |

`text`·`media_ref`·`source`·`id` 등 그 밖의 키는 들고 다녀도 되며 **랭커가 무시한다**
(원칙 2: 검색 결정은 결정론적 산수뿐, 내용은 점수에 안 닿음). `store.search()`가
돌려주는 dict가 이미 이 계약을 만족한다. 강제·증명은 `tests/test_rank.py`의
`test_ranker_ignores_modality` / `test_mixed_modality_pool_ranks_by_score`.

## 왜 활성도 의미가 그대로 전이되나
"잊힌 가치" 신호는 날짜·볼륨 기반이라 모달리티와 무관하다. 오래 묵힌 사진, 뒤에
올린 영상들에 덮인 옛 클립 — 전부 도구가 끌어올리려는 바로 그 "잊힌 것"이다. 점수식의
의미를 바꿀 필요가 없다.

## 미래 배선 (아직 구현 안 함 — 방향만 기록)
사용자가 택한 검색 모델: **모달리티별 공간 + 질의별 인코더.**

- **텍스트 → 텍스트**: 지금 그대로. KURE로 질의 임베딩 → 텍스트 vec 테이블 검색.
  (KURE의 한국어 강점 유지)
- **텍스트 → 이미지**: *별도* 파이프라인. CLIP계(예: SigLIP) **공유 텍스트/이미지
  공간**에서, 같은 텍스트 질의를 CLIP-텍스트 인코더로 임베딩 → 이미지 vec 테이블
  (CLIP-이미지로 인덱싱) 검색.
- 두 검색이 각각 `Candidate` 목록을 내놓고 → **하나의 `rank_*` 호출로 합친다.**
  `store.py`는 이미 모델/공간별 vec 테이블을 분리하므로 이 구조가 자연스럽게 확장된다.

```
질의 ──┬─ KURE 인코딩 ─────▶ vec_kure(텍스트) ─┐
       └─ CLIP-텍스트 인코딩 ─▶ vec_clip(이미지) ─┴─▶ [Candidate…] ─▶ rank_* ─▶ 결과
```

## 합칠 때 남는 설계 질문 (나중에 결정)
- **볼륨 축의 범위**: `newer_counts`를 모달리티 *안에서* 셀지(이미지는 이미지끼리),
  *교차*로 셀지(옛 사진이 이후 글들에 덮였다고 볼지). 의미가 달라지는 지점.
- **풀 병합/인터리빙**: 서로 다른 임베딩 공간의 `distance`는 스케일이 다를 수 있다.
  밴드패스는 *질의별 분포의 퍼센타일*이라 공간이 섞이면 분포가 왜곡될 수 있으니,
  모달리티별로 밴드패스를 따로 돌린 뒤 점수를 합치는 편이 안전할 가능성.
- **결과 표시**: 한 줄로 섞을지, 모달리티별 섹션으로 나눌지.
- **밴드패스 스케일**: 밴드패스는 풀 전수 분포를 가정함. 수만 항목 매체 진입 시
  retrieve를 top-N 샘플 분포로 교체 필요.

이 질문들은 실제 2번째 모달리티를 붙일 때 다룬다. 지금 단계의 핵심은 *계약이 명시되어
있어 그때 뇌를 건드릴 필요가 없다*는 것.
